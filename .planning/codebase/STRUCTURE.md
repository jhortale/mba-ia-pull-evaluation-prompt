## Directory Purposes

**`src/`**
- Purpose: Production code — entry-point scripts and reusable libraries
- Contains: 4 executable Python scripts + 2 utility modules
- Key files: `evaluate.py` (orchestrator), `metrics.py` (scorer), `utils.py` (shared abstractions)
- Invocation: Run scripts directly from repo root: `python src/evaluate.py`
- Exports: All modules importable for testing; metrics.py has smoke test in `__main__`

**`tests/`**
- Purpose: Pytest validation suite
- Contains: Single test file with 6+ test methods
- Key file: `test_prompts.py`
- Invocation: `pytest tests/test_prompts.py -v`
- Path setup: Adds `src/` to sys.path at line 10 for import resolution

**`prompts/`**
- Purpose: YAML-based prompt versions for Hub publishing
- Contains: Prompt template files with system_prompt, user_prompt, metadata
- Naming: `bug_to_user_story_v{N}.yml` where v1 is given (low-quality), v2+ are optimizations
- Key file: `bug_to_user_story_v1.yml` (read-only, shows example of poor prompt)
- Usage: Pulled from/pushed to LangSmith Hub (Hub is source of truth for published versions)
- User-editable: v2.yml must be created and edited by user

**`datasets/`**
- Purpose: Evaluation examples in JSONL format
- Contains: Single file with 15 example bug reports + reference user stories
- Key file: `bug_to_user_story.jsonl` (static, not modified during evaluation)
- Format: One JSON object per line; each line has `inputs`, `outputs`, `metadata` keys
- Coverage: 5 simple, 7 medium, 3 complex bugs from domains (e-commerce, SaaS, mobile, ERP, CRM)

**`.planning/codebase/`**
- Purpose: GSD (Getting Stuff Done) documentation
- Contains: Architecture, structure, conventions, testing, concerns, integrations docs
- Generated by: `/gsd-map-codebase` command
- Consumed by: `/gsd-plan-phase` and `/gsd-execute-phase` (future phases reference these)

**Root**
- `requirements.txt`: 19 pinned dependencies (langchain, langsmith, langchain-openai, langchain-google-genai, python-dotenv, pyyaml, pydantic, pytest)
- `.env.example`: Template showing all required env vars (no secrets, just var names and format)
- `README.md`: Challenge specification, execution order, technology list, delivery requirements

**Configuration:**
- `.env.example` — Template for all required environment variables

**Core Logic:**

- `src/utils.py` — Shared abstractions (240+ lines): `get_llm()`, `get_eval_llm()`, `extract_json_from_response()`, YAML I/O, validation

**Testing:**
- `tests/test_prompts.py` — 6+ pytest test methods validating prompt structure and content

**Data (Static):**
- `prompts/bug_to_user_story_v1.yml` — Given low-quality prompt (shows what NOT to do)
- `datasets/bug_to_user_story.jsonl` — 15 evaluation examples (never modified)

## Naming Conventions

**Files:**
- Scripts: `snake_case` ending in `.py` (e.g., `pull_prompts.py`, `test_prompts.py`)
- YAML prompts: `{purpose}_v{N}.yml` where purpose is task name, N is version (e.g., `bug_to_user_story_v1.yml`)
- Data: `{dataset_name}.jsonl` (e.g., `bug_to_user_story.jsonl`)
- Directories: `snake_case` (e.g., `src/`, `tests/`, `prompts/`, `datasets/`)

**Python Functions:**
- Entry points: `main()` (pattern: all CLI scripts have `main()` + `if __name__ == "__main__": sys.exit(main())`)
- LLM factories: `get_llm()`, `get_eval_llm()`
- Validators: `validate_prompt()`, `validate_prompt_structure()`, `check_env_vars()`
- Data loaders: `load_yaml()`, `load_dataset_from_jsonl()`
- Savers: `save_yaml()`
- Metric functions: `evaluate_{metric_name}()` (e.g., `evaluate_f1_score()`, `evaluate_clarity()`)
- Formatters: `format_score()`, `print_section_header()`
- Helpers: `extract_json_from_response()`, `pull_prompt_from_langsmith()`

**Python Classes:**
- Test class: `TestPrompts` in `test_prompts.py` (pytest convention)

**LangSmith Hub Naming:**
- Prompt names on Hub: `{username}/bug_to_user_story_v{N}` (e.g., `joao.marcelo/bug_to_user_story_v2`)
- Dataset names in LangSmith: `{project_name}-eval` (e.g., `prompt-optimization-challenge-resolved-eval`)
- Project name (tracing): `prompt-optimization-challenge-resolved` (default from `LANGSMITH_PROJECT` env var)

**Environment Variables:**
- LangSmith config: `LANGSMITH_*` (LANGSMITH_API_KEY, LANGSMITH_PROJECT, LANGSMITH_ENDPOINT, LANGSMITH_TRACING)
- User identity: `USERNAME_LANGSMITH_HUB` (not LANGSMITH_USERNAME — HUB suffix)
- LLM provider selection: `LLM_PROVIDER` (values: "openai" or "google")
- LLM model names: `LLM_MODEL` (e.g., "gpt-4o-mini"), `EVAL_MODEL` (e.g., "gpt-4o")
- API keys: `OPENAI_API_KEY`, `GOOGLE_API_KEY` (both optional, one required based on LLM_PROVIDER)

## Where to Add New Code

**New Metric Function:**
- Add function to `src/metrics.py` following pattern: `evaluate_{metric_name}(question, answer, reference) -> Dict[str, Any]`
- Function must call `get_eval_llm()`, construct judge prompt, call LLM, extract JSON, return dict with at least `score` and `reasoning`
- Import new function in `src/evaluate.py` if it should be included in main pipeline
- Update README metric list if needed

**New CLI Script (e.g., validate prompts locally):**
- Create `src/{script_name}.py` with `main()` function and `if __name__ == "__main__": sys.exit(main())`
- Import reusable helpers from `utils.py` and `metrics.py`
- Follow pattern: check env vars → load data → process → output
- User runs: `python src/{script_name}.py`

**New Validation Test:**
- Add test method to `TestPrompts` class in `tests/test_prompts.py`
- Method name must start with `test_` (pytest convention)
- Use `load_prompts()` helper to load YAML
- Call `validate_prompt_structure()` from utils if validating structure, or write custom assertion
- Run: `pytest tests/test_prompts.py::TestPrompts::{method_name} -v`

**Shared Utility Function:**
- Add to `src/utils.py` if it's used by multiple modules
- Keep utils focused on abstractions (LLM, I/O, validation), not business logic
- Document with docstring including Args, Returns, Raises
- If a new dependency is needed, add to `requirements.txt` and update setup docs

**New Dataset Example:**
- Add line to `datasets/bug_to_user_story.jsonl` as single JSON object with structure: `{"inputs": {"bug_report": "..."}, "outputs": {"reference": "..."}, "metadata": {...}}`
- Metadata must include: `domain`, `type`, `complexity`
- User (in tests): loads entire file with `load_dataset_from_jsonl()`
- Note: Dataset is static; only use for extending evaluation scope

## Execution Context

**Current Working Directory (CWD):**
- All scripts are run from repo root: `cd /path/to/mba-ia-pull-evaluation-prompt && python src/evaluate.py`
- CWD-relative paths in code (e.g., `datasets/bug_to_user_story.jsonl`) resolve to repo root
- Tests also run from repo root: `cd /path/to/mba-ia-pull-evaluation-prompt && pytest tests/test_prompts.py`
- sys.path setup: In `tests/test_prompts.py`, explicitly adds `src/` to path with `sys.path.insert(0, str(Path(__file__).parent.parent / "src"))` (line 10)

**Module Imports:**
- Within `src/`: relative imports not used; all modules import from utils as if sibling
- Example in evaluate.py: `from utils import check_env_vars, format_score, print_section_header, get_llm as get_configured_llm`
- Works because when run as `python src/evaluate.py`, the `src/` directory is in sys.path implicitly

**Execution Order (Typical User Flow):**
1. User configures `.env` (creates file, fills in credentials)
2. User runs `python src/pull_prompts.py` → downloads v1.yml from Hub
3. User edits `prompts/bug_to_user_story_v2.yml` manually (applies prompt engineering)
4. User runs `python src/push_prompts.py` → publishes v2.yml to Hub
5. User runs `python src/evaluate.py` → pulls v2 from Hub, scores on dataset, reports results
6. If scores < 0.9: user edits v2.yml again → repeat steps 4-5 (iteration cycle)
7. Once all metrics >= 0.9: user runs `pytest tests/test_prompts.py` → validates tests pass
8. User commits and pushes to GitHub
